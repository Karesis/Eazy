Eazy 语言特性详解文档 v1.0文档状态: 初稿 (基于截至 2025-04-28 的讨论)1. 引言Eazy 是一门处于积极设计阶段的新兴编程语言。其核心设计哲学是 “万物皆块 (Everything is a Block)”，旨在通过统一的“块”概念来构建语言的全部结构和操作。Eazy 的一个关键设计目标是弥合高级语言与低级语言（如汇编）之间的鸿沟，提供一种既具备高级语言结构化、抽象化能力，又能进行精细底层控制（如直接映射汇编指令）的编程体验。这使得 Eazy 具备应用于系统级编程（例如内核开发）的潜力。本文档旨在详细阐述 Eazy 当前已定义的各项核心特性，包括其语法、语义、设计原理以及相互之间的关系。2. 核心哲学与基础概念块 (Block):定义: 语言的最基本结构单元，通过 @名称: 语法定义。每个 @ 块代表一个独立的模板或代码单元。@main: 是预定义的程序入口块。本质: @ 块不仅是代码容器，更是一种可被实例化的“模板”。它们封装了特定的行为逻辑和潜在的状态。容器 (Container):定义: Eazy 中存储数据的基本抽象。从简单的标量值（如 int）到复杂的结构（struct），都被视为不同类型和大小的“容器”。声明即创建: 声明一个容器（如 int a 或 struct point {int x, int y}）不仅仅是指定类型和名称，更是直接创建了一个具有该名称和指定容量的存储实例。作用域 (Scope):界定: 主要由 @ 块定义。一个 @ 块为其内部所有内容（行块、标签块、嵌套 @ 块）提供了一个共享的作用域。访问规则: 在同一作用域内声明的容器可以被该作用域内的所有代码（遵循特定访问限制）访问。3. 特性详解3.1 块定义、绑定 (.) 与镜像 (image)块定义: @名称: 定义了一个块模板。行块 (Lineblock): 块内部由一系列单行语句构成，称为行块。它们是执行具体操作的基本单位，需要统一缩进。本质上，行块是匿名的“内块”。绑定 (.): 在块内部，可以使用 .名称 语法（如 .result）将某个内部容器的值标记为“已绑定”。这并不会改变容器本身，而是声明该值应包含在通过 image 创建的镜像中。一个块可以绑定多个值。镜像创建 (image):语法: image 镜像名 模板块名过程: 该指令触发对“模板块名”所指定 @ 块内部逻辑的一次完整执行。重要的是，这个执行过程是非侵入式的，它不会改变原始模板块的状态，也不会“进入”原始块的执行上下文。结果: 执行完成后，生成一个名为“镜像名”的新实体。这个“镜像”是一个结构化的状态快照，它捕获了该次执行结束时所有被 .名称 绑定的内部容器的值。访问: 通过 镜像名.绑定名 (如 block_add.result) 来访问镜像中捕获的特定绑定值。设计意图: image 提供了一种安全、隔离的方式来执行块逻辑并获取其计算结果（可能包含多个值）。它强调块的封装性和模板性，类似于创建了一个包含执行结果的对象实例。示例:@calculator:
  int operand1
  int operand2
  set operand1 10
  set operand2 5

  int sum_res
  set sum_res operand1 + operand2
  .sum  # 绑定加法结果

  int diff_res
  set diff_res operand1 - operand2
  .difference # 绑定减法结果

@main:
  image calc_snapshot calculator # 创建镜像
  print calc_snapshot.sum         # 访问镜像中的 sum (输出 15)
  print calc_snapshot.difference  # 访问镜像中的 difference (输出 5)
  exit
3.2 参数化块与参数传递定义: 块可以在定义时声明参数，语法为 @名称(类型1 参数名1, 类型2 参数名2, ...):。这极大地增强了块的可复用性。通过 image 传递: image 镜像名 模板块名(值1, 值2, ...)。传递机制:这不是传统意义上的传值或传引用。当创建镜像时，传入的“值1”、“值2”等被用作块内部对应参数名（“参数名1”、“参数名2”）所代表的容器的初始值。执行依然发生在隔离的镜像创建过程中。类型提示: 参数定义中的类型（如 int x）主要用于指导块内部的操作（如 set result x+y 中 + 操作应按整数处理 x 和 y），不代表在 image 创建时进行严格的静态类型检查。类型符合性可能在具体操作执行时才确定。示例:@adder(int first, int second): # 定义带参数的块
  int calculation_result
  set calculation_result first + second # 使用参数容器进行计算
  .result                         # 绑定结果

@main:
  int val_a
  int val_b
  set val_a 7
  set val_b 8
  image my_adder_instance adder(val_a, val_b) # 传递 a, b 的值作为 first, second 的初始值
  print my_adder_instance.result             # 输出 15
  exit
3.3 调用 (call) 与返回 (ret)目的: 提供一种与 image 截然不同的块交互方式，侧重于直接执行块并获取一个单一的、明确指定的返回值。语法: call 结果容器 目标块名(参数...)。ret 语句:被 call 的块必须在其执行路径中包含至少一个 ret 值 语句。ret 语句立即终止当前块的执行，并将指定的“值”作为 call 操作的结果返回。如果在 call 执行流中没有遇到 ret 语句（例如执行到块末尾），将产生运行时错误。执行模型:call 是一种侵入式操作，它直接将控制流转移到“目标块名”的起始处开始执行（类似于底层 goto 或过程调用）。执行发生在当前的上下文中，不是在隔离环境中。在 call 执行期间，目标块内部的任何 .绑定 语句都会被完全忽略，因为 call 的目的不是创建状态快照。准入条件 (准入条件):在执行 call 之前，系统会检查调用时提供的参数数量和“格式”（可能指结构或占位符）是否与目标块定义的参数列表匹配。关键： 此检查不涉及对传入参数值的类型进行验证。类型兼容性问题可能在块内部实际使用参数进行运算时才暴露。设计意图: call 提供了更接近传统函数调用的交互模式，适用于需要单一明确结果且允许直接控制流转移的场景。它与 image 形成互补。示例:@multiply(int factor1, int factor2):
  int product
  set product factor1 * factor2
  .ignored_binding # 这个绑定在 call 时无效
  ret product      # 必须有 ret 来返回值

@main:
  int x
  int y
  set x 6
  set y 7
  int multiplication_result

  call multiplication_result multiply(x, y) # 直接调用并获取单一返回值
  print multiplication_result              # 输出 42
  exit
3.4 标签块、内块与 goto 控制流内块 (Inner Block): 泛指位于 @ 块内部、不以 @ 开头定义的代码单元。它们与父 @ 块共享作用域。标签块 (Label Block):定义: 使用 名称: 语法在 @ 块内部定义，例如 loop_start:。功能: 作为 goto 指令的跳转目标，标记代码中的特定位置。特性: 是内块的一种；没有 @ 块的隔离性；不能被 image 或 call；可以为空（如 loop_end:）。匿名内块 (Anonymous Inner Block):所有常规的单行语句（行块，如 int a, set a 10, print a）本质上都是匿名的内块。因为没有名称，它们不能作为 goto 的目标。goto 指令:语法: goto 标签名行为: 执行无条件跳转到同一 @ 块内部、同一层级“野代码”区域中指定的“标签名”位置。严格限制: goto 的使用范围非常受限，不能用于跳转进出 @ 块（无论是嵌套块还是父块），也不能跳转到不同层级的野代码。它基本上只用于实现同一段连续代码（“野代码”区域）内的局部循环和跳转。设计意图: 提供基础的块内控制流能力。严格的限制旨在避免滥用 goto 导致的“面条代码”，同时维护 @ 块的结构化和封装性。示例: (同之前的循环示例)@main:
  int counter
  set counter 3

  loop_entry:      # 标签块
    if counter = 0
      goto loop_exit # 跳转到同级标签

    print counter
    sub counter 1
    goto loop_entry # 跳转到同级标签

  loop_exit:       # 空标签块

  exit
# 输出: 3 2 1
3.5 嵌套块 (块内的块)定义: 允许在一个 @ 块（父块）内部使用 @名称: 语法定义另一个 @ 块（嵌套块/子块）。作用域关系 (共享与隔离并存):共享: 嵌套块完全共享其父块的作用域，可以直接读取和修改父块中定义的容器。访问不对称 (封装性): 这是关键规则。父块的“野代码”区域不能直接访问嵌套块的内部实现细节（例如，不能直接读写嵌套块内部声明的容器，除非该容器被绑定并通过镜像访问）。父块与嵌套块的主要交互方式是通过 image 或 call。父块的野代码不能使用 goto 跳转进入嵌套块。相对独立性: 尽管共享作用域，嵌套块在执行逻辑和内部状态上保持相对独立，其实现对父块的野代码是隐藏的。术语: @ 块（无论顶层或嵌套）称为主块 (主块)。相对于一个主块，其内部的所有内容（行块、标签块、嵌套主块）都可视为从块 (从块)。设计意图: 提供模块化和代码组织能力。允许将相关逻辑封装在嵌套块中，同时又能方便地访问外部上下文（父块作用域），并通过受控接口 (image/call) 与外部交互。示例:@outer:
  int shared_value
  set shared_value 10

  @inner:             # 嵌套块定义
    int inner_result
    # 可以直接访问父块的 shared_value
    set inner_result shared_value * 2
    .result          # 绑定自己的结果

  # 父块通过 image 与嵌套块交互
  image inner_snapshot inner
  print inner_snapshot.result # 输出 20

  exit
3.6 嵌套交互规则与权级 (权级)权级概念: 虽然父块内的所有直接从块（野代码、嵌套 @ 块）在作用域上共享，但在控制流交互上存在隐式的“权级”差异：野代码（行块、标签块）的权级被视为低于嵌套的 @ 主块。交互规则总结:goto:父块野代码 goto -> 嵌套块内部: 非法 (权级不足/跨越边界)。嵌套块 goto -> 父块野代码标签: 非法 (跨越边界/封装性)。(重申) goto 基本仅限于在同一块的同级野代码区域内跳转。call:父块野代码 call -> 嵌套块: 允许 (遵循 call 规则)。嵌套块 call -> 同父块下的兄弟嵌套块: 允许 (同层级交互)。嵌套块 call -> 兄弟块内部的嵌套块 (侄子块):直接 call Nephew: 非法。原因：Nephew 的执行可能依赖其直接父块（Uncle）的野代码所建立的上下文。直接调用会缺少此上下文。通过.路径 call Uncle.Nephew: 允许。这里的 . 不仅仅是名称分隔符，它隐含了一个执行语义：在最终 call Nephew 之前，必须先模拟或实际执行 Uncle 块的野代码部分，以正确建立 Nephew 运行所需的（来自 Uncle 的）作用域上下文。这个机制确保了调用的有效性。设计意图: 这些规则旨在平衡嵌套带来的便利性（共享作用域）与结构化编程的原则（封装、受控交互）。. 路径调用机制是一个独特的解决方案，用于在保持封装的同时，安全地调用深层嵌套的、依赖其直接父上下文的块。示例 (. 路径调用概念):@grandparent:
  int gp_val
  set gp_val 1

  @parent1:
    int p1_val
    set p1_val gp_val + 10 # p1_val = 11

    @child1:
      int c1_val
      set c1_val p1_val * 2 # c1_val = 22
      ret c1_val

  @parent2:
    int p2_val
    set p2_val gp_val + 100 # p2_val = 101

    int result_container
    # 假设需要调用 parent1 内部的 child1
    # 直接 call child1 是非法的
    # 需要通过路径调用以建立 parent1 的上下文
    call result_container parent1.child1

    print result_container # 应该输出 22

  call dummy parent2 # 执行 parent2
  exit
(注意: 上述.路径调用示例基于对规则的理解推演，具体语法和实现细节待定)3.7 超定义 (struct) - 结构化容器实例语法: struct 实例名 { 类型1 成员名1, 类型2 成员名2, ... }核心本质: 这条语句不是在定义一个可复用的“类型”或“类”。它直接声明并创建了一个具体的、单一的、名为“实例名”的复合容器实例。类比: 如果 int a 是创建一个名为 a 的简单 int 容器实例，那么 struct point {int x, int y} 就是创建一个名为 point 的复合容器实例，这个实例内部包含两个名为 x 和 y 的 int 小容器。访问: 使用 实例名.成员名 的点号表示法来访问和操作其内部的成员容器。设计意图: 提供一种简单直接的方式来组织相关数据，符合“万物皆容器”的理念。它避免了传统类型系统中的“类型定义”与“实例创建”的分离步骤，使得结构化数据的创建更加直接。示例:@main:
  # 直接创建名为'player_data'的复合容器实例
  struct player_data {
    int health,
    int score,
    char initial
  }

  # 直接访问并设置其成员
  set player_data.health 100
  set player_data.score 0
  set player_data.initial 'J' # 假设 char 类型和赋值

  # 使用其成员
  print player_data.health
  sub player_data.score 10 # 假设 score 可以被修改
  print player_data.score

  exit
3.8 内置类型Eazy 语言预计会提供一组基础的容器类型，作为构建更复杂结构的基础。目前已提及的包括：int (整数)char (字符)未来可能会根据语言发展需要添加更多内置类型（例如，浮点数、布尔值、指针/地址类型等，特别是考虑到系统编程目标）。4. 总结与展望Eazy 语言展现了一种独特且高度统一的设计哲学。通过将“块”作为核心抽象，并围绕它设计了 image (状态快照) 和 call (直接执行) 两种交互机制，以及嵌套、作用域、受限控制流和直接结构化容器实例 (struct) 等特性，Eazy 构建了一个自洽且富有潜力的编程模型。其明确的系统编程目标（统一高低级体验、用于内核开发）为其设计带来了挑战，也注入了强大的驱动力。如何在提供结构化便利（“Eazy”）的同时，确保足够的底层控制力、性能可预测性和对底层硬件的透明映射，将是 Eazy 未来发展需要持续探索和解决的关键问题。当前定义的特性集为 Eazy 奠定了坚实的基础。后续可能需要进一步考虑内存管理模型、并发机制、更丰富的类型系统、错误处理机制以及与外部世界（如操作系统、硬件）的交互接口等，以支撑其宏伟的设计目标。这份更详细的文档希望能更好地满足你的需求。请随时提出反馈或需要进一步澄清的地方！
